# 4주차. 리스트 (List)와 추상자료형 (ADT)

<br>

## 1\. 추상자료형 (ADT, Abstract Data Type)

프로그래밍에서 데이터구조를 설계할 때, 복잡한 내부 구현 코드는 철저히 숨기고 **"이 구조는 어떤 데이터들을 담고, 밖에서 어떤 기능(작업)들을 쓸 수 있는지" 그 껍데기(명세, Specification)만 정의해 둔 것**을 추상자료형(ADT)이라고 한다.

- **ADT가 명세(정의)하는 3가지 요소**
  1.  저장될 **데이터**의 형태
  2.  데이터에 대해 수행할 수 있는 \*\*작업(메소드)\*\*들
  3.  작업 중 발생할 수 있는 \*\*에러 상황(예외)\*\*들
- **예시 (주식거래 시스템 모델링)**
  - 데이터: buy/sell 주문들
  - 작업: `order_buy()`, `order_sell()`, `cancel()`
  - 에러 상황: 존재하지 않는 주식을 사려고 하거나, 없는 주문을 취소하려 할 때 에러 발생.

<br>

## 2\. 리스트 ADT (List ADT)

리스트 ADT는 순서가 있는 임의의 개체(원소)들의 모음을 모델링한 것이다.
배열의 '인덱스'와 비슷한 개념으로, 리스트에서는 **순위(Rank)**라는 도구를 사용해 원소에 접근한다.

- **순위(Rank)의 정의**: 어떤 원소의 순위란, '**그 원소 앞에 있는 원소들의 개수**'를 뜻한다. (가장 첫 원소의 순위는 0, 두 번째는 1이 된다.)

### 2-1. 순위 기반 리스트의 모든 메소드

- **상태 확인 메소드**
  - `size()`: 리스트에 들어있는 총 원소의 개수를 반환한다.
  - `isEmpty()`: 리스트가 비어있으면 `true`, 아니면 `false`를 반환한다.
- **접근 및 수정 메소드**
  - `get(r)`: 순위 $r$에 있는 원소를 반환한다.
  - `set(r, e)`: 순위 $r$에 있는 원소를 새로운 원소 $e$로 교체하고, **기존에 있던 원소를 반환**한다.
- **삽입 및 삭제 메소드**
  - `add(r, e)`: 순위 $r$ 위치에 새로운 원소 $e$를 삽입한다. $r$번부터 그 뒤에 있던 모든 원소들은 순위가 1씩 뒤로 밀린다.
  - `remove(r)`: 순위 $r$에 있는 원소를 삭제 및 반환한다. 빈자리를 채우기 위해 뒤에 있는 원소들의 순위가 1씩 앞으로 당겨진다.

### 2-2. 발생할 수 있는 3가지 예외 (Exceptions)

자료구조를 안전하게 설계하기 위해, 잘못된 요청이 들어오면 아래의 3가지 예외(에러)를 발생시켜야 한다.

1.  **`InvalidRankException` (유효하지 않은 순위 예외)**
    - 발생 조건: $r < 0$ 이거나 $r \ge n$ (단, `add`의 경우 $r = n$까지 허용)인 상황에서 `get`, `set`, `remove` 등을 호출할 때 발생한다.
2.  **`EmptyListException` (빈 리스트 예외)**
    - 발생 조건: 리스트가 완전히 비어있는 상태(`size() == 0`)에서 원소를 꺼내거나(`get`) 지우려고(`remove`) 할 때 발생한다.
3.  **`BoundaryViolationException` (경계 침범 예외)**
    - 발생 조건: 연결리스트 등에서 첫 번째 노드 이전으로 가려고 하거나(예: Head의 `prev` 접근), 마지막 노드 다음으로 넘어가려고(예: Tail의 `next` 접근) 할 때 발생하는 치명적인 에러다.

## 3\. 리스트 ADT의 구현: 배열 vs 단일 연결리스트

리스트 ADT라는 '설계도(규격)'를 실제 코드로 구현하는 방법은 크게 2가지가 있다. 이 둘은 성능(시간 복잡도)에서 극명한 차이를 보인다.

### 3-1. 배열 기반 구현 (Array-based Implementation)

크기 $N$짜리 배열 $V$를 만들고, 실제 들어있는 원소 개수를 $n$이라고 둔다.

- **`get(r)` / `set(r, e)` $\rightarrow \boldsymbol{O(1)}$ 시간**
  - 배열은 시작 주소만 알면 인덱스로 즉시 찾아갈 수 있으므로 가장 빠르다.
- **`add(r, e)` $\rightarrow$ 최악의 경우 $\boldsymbol{O(n)}$ 시간**
  - $r$ 자리에 새 원소를 넣기 위해, $n-1$번(맨 끝) 원소부터 $r$번 원소까지 차례대로 뒤로 한 칸씩 이사(Shift)시켜야 한다.
  - 최악의 경우(맨 앞 $r=0$에 넣을 때) 전체 $n$개를 모두 옮겨야 하므로 $O(n)$이 걸린다.
  - **오류 처리**: 만약 $n=N$ (배열이 꽉 참) 상태에서 `add`를 부르면 에러가 발생한다. (이를 해결하기 위해 꽉 차면 더 큰 배열을 새로 할당받아 이사 가는 '동적 배열(Dynamic Array)' 기법을 쓰기도 한다.)
- **`remove(r)` $\rightarrow$ 최악의 경우 $\boldsymbol{O(n)}$ 시간**
  - $r$번 자리를 지우고 난 뒤, 뻥 뚫린 구멍을 메우기 위해 $r+1$번부터 $n-1$번까지의 원소들을 차례대로 앞으로 한 칸씩 당겨와야(Shift) 하므로 $O(n)$이 걸린다.

### 3-2. 단일 연결리스트 기반 구현 (Singly Linked List-based)

원소들을 단일 연결리스트의 노드들에 하나씩 담아서 연결한다.

- **`get(r)` / `set(r, e)` $\rightarrow$ 최악의 경우 $\boldsymbol{O(n)}$ 시간**
  - 배열처럼 한 번에 찾아갈 수 없다. 무조건 Head(맨 앞)부터 시작해서 링크를 타고 $r$번을 징검다리 건너듯 이동해야 하므로, 맨 끝을 찾을 때는 $O(n)$이 걸린다.
- **`add(r, e)` / `remove(r)` $\rightarrow$ 최악의 경우 $\boldsymbol{O(n)}$ 시간**
  - 포인터를 끊고 연결하는 삽입/삭제 행위 자체는 1초($O(1)$)면 끝난다.
  - **하지만\!** 삽입/삭제를 할 위치인 $r-1$번 노드를 찾아가는 데(탐색) 시간이 걸리기 때문에 결국 전체 수행 시간은 최악의 경우 $O(n)$이 된다.

### 배열 vs 연결리스트 성능 요약

| 메소드                   | 배열 (Array)       | 단일 연결리스트 (Singly Linked List) |
| :----------------------- | :----------------- | :----------------------------------- |
| `size()`, `isEmpty()`    | $O(1)$             | $O(1)$                               |
| `get(r)`, `set(r, e)`    | **$O(1)$**         | $O(n)$                               |
| `add(r, e)`, `remove(r)` | $O(n)$ (이사 비용) | $O(n)$ (찾아가는 비용)               |

## 4\. 리스트 ADT의 확장: 이중 연결리스트와 위치(Position) 기반 접근

단일 연결리스트는 뒤로 갈 수 없다는 한계가 있고, 순위($r$) 기반으로 삽입/삭제를 하면 매번 노드를 처음부터 찾아가느라 $O(n)$이라는 낭비가 발생한다. 이를 극복하기 위한 확장 기법이다.

### 4-1. 이중 연결리스트와 더미 노드 (Dummy Nodes)

- 양방향 통행이 가능한 **이중 연결리스트(Doubly Linked List)**를 사용한다.
- **헤더(Header)**와 **트레일러(Trailer)**라는 텅 빈 가짜 노드(더미 노드)를 리스트의 맨 앞과 맨 뒤에 둔다.
- 이를 통해 데이터가 아예 없거나, 맨 앞/맨 뒤에 데이터를 삽입/삭제할 때 예외 처리를 해야 하는 번거로움을 완전히 없애준다 (모든 실제 노드는 항상 앞뒤에 노드를 가지게 됨).

### 4-2. 위치(Position) 기반 삽입/삭제 로직 $\rightarrow \boldsymbol{O(1)}$ 달성

순위($r$) 번호를 주는 대신, 특정 노드의 메모리 주소(위치)를 직접 알려주면 찾아가는 시간 $O(n)$을 없애고 **즉시 $O(1)$에 삽입/삭제가 가능**해진다.

1.  **`addNode(p, e)` 로직 (삽입)**: 노드 $p$의 바로 뒤에 새로운 원소 $e$를 가진 노드 $q$를 끼워 넣는다.
    - 새 노드 $q$ 생성, 값 $e$ 저장
    - $q$의 `prev`가 $p$를 가리키게 함
    - $q$의 `next`가 $p$의 기존 다음 노드($v$)를 가리키게 함
    - $v$(기존 다음 노드)의 `prev`가 $q$를 가리키게 수정
    - $p$의 `next`가 $q$를 가리키게 수정 (포인터 4개 갱신 끝 $\rightarrow O(1)$)
2.  **`removeNode(p)` 로직 (삭제)**: 노드 $p$를 리스트에서 떼어낸다.
    - $p$의 앞 노드(`u`)를 찾음
    - $p$의 뒤 노드(`w`)를 찾음
    - $w$의 `prev`가 $u$를 가리키게 수정 (건너뜀)
    - $u$의 `next`가 $w$를 가리키게 수정 (건너뜀)
    - (포인터 2개 갱신 끝 $\rightarrow O(1)$)

- **추가된 메소드**: `first()`, `last()`, `prev(p)`, `next(p)`
- **위치 기반 삽입/삭제**: `addBefore(p, e)`, `addAfter(p, e)`, `addFirst(e)`, `addLast(e)`, `remove(p)`
- **구현 핵심 (더미 노드가 있는 이중 연결리스트)**:
  - 앞뒤로 이동 가능한 **이중 연결리스트**를 사용하고, 맨 앞과 뒤에 데이터가 없는 가짜 노드인 **헤더(Header)와 트레일러(Trailer)**를 붙인다.
  - 이를 통해 예외 처리(빈 리스트 삽입 등)를 무시할 수 있으며, 타겟 노드 $p$의 위치를 직접 받으므로 **모든 삽입/삭제 연산을 탐색 없이 즉시 $O(1)$** 만에 처리할 수 있게 된다\!

## 5\. 리스트의 확장: 그룹(Group)과 공유(Share)

단순히 데이터를 한 줄로 늘어놓는 것을 넘어, 데이터 간의 복잡한 소속 관계를 표현하기 위한 심화 구조다.

### 5-1. 그룹화 (Grouping)

- **정의**: 데이터 원소들이 각각 상이한 '그룹(카테고리)'에 속하며, 그룹마다 크기가 다른 형태다. (예: 쇼핑몰의 상품들을 채소, 전자제품, 육류 그룹으로 분류하거나, 대학의 강좌를 교수님별로 묶는 것)
- **구현 방법 (레코드의 리스트 사용)**
  - 원소를 저장하는 노드 구조체(레코드) 내부에 `elem(데이터)` 필드뿐만 아니라 **`group(소속 그룹명)` 필드를 추가**하여 리스트로 연결한다.
  - **장점**: 구조가 매우 직관적이고 단순하다.
  - **단점**: "특정 그룹(예: 채소)에 속한 데이터만 다루고 싶다"고 할 때, 관련 데이터들끼리 묶여있지 않으므로 결국 **리스트 전체를 처음부터 끝까지 다 뒤져봐야 하는 비효율**이 발생한다.

### 5-2. 공유 (Sharing)

- **정의**: 데이터 원소들이 상이한 여러 그룹에 의해 '동시에 공유'되는 현상이다. (예: 대학생 A그룹과 B그룹이 공통된 교양 과목을 수강하는 경우)

- **데이터 복제 금지의 원칙**: 공유된다고 해서 각 그룹 리스트에 데이터를 복제해서 넣으면(A그룹에도 넣고 B그룹에도 복사해 넣으면) 메모리가 크게 낭비되고, 정보 갱신 시 충돌이 나므로 **데이터 중복(복제)은 절대 허용하지 않는다.**

- **구현 방법 1: 포인터 참조 (고립형)**
  - 원소 데이터들을 메모리 한구석에 한 번만 저장해 두고, 각 그룹에서는 그 데이터가 있는 메모리 주소(포인터)만 가리키게 한다.
  - **장/단점**: 단순하고 기억장소 낭비가 적지만, "이 원소가 지금 어떤 그룹들에게 공유되고 있지?"를 파악하려면 전체를 뒤져야 하므로 상호 교차 점검이 불가능하다.

- **구현 방법 2: 교차점 서브리스트 (Multi-Linked List, 다중 연결리스트) - 가장 권장됨**
  원소와 그룹의 상호 관계를 직접 매핑하는 방식이며, 두 가지 접근법이 있다.
  1.  **2차원 Boolean 배열**: 행(Row)을 원소, 열(Column)을 그룹으로 두어 속해있으면 `True(1)`로 저장한다. 속도가 매우 빠르지만, 빈칸이 많을 경우 메모리 낭비가 심하다.
  2.  **두 개의 배열과 원형 헤더 연결리스트**:
      - 원소들을 담아두는 배열(Elements)과 그룹들을 담아두는 배열(Groups) 2개를 별도로 만든다.
      - 어떤 가입자 $g$가 쿠폰 $e$를 보유(공유)하게 되면, 두 배열을 교차시키는 거대한 다중 연결리스트 내부에 $(e, g)$라는 노드를 생성하여 엮는다.
      - **장점**: 가로로 링크를 타면 특정 그룹이 가진 모든 원소를 볼 수 있고, 세로로 링크를 타면 특정 원소를 가진 모든 그룹을 바로 확인할 수 있어 **양방향 다중 검색에 극도로 효율적**이며 메모리 낭비도 최소화된다.

## 6\. 응용문제: 요세푸스(Josephus) 촛불 끄기 문제

- **상황**: 원탁에 1번부터 $n$번까지의 촛불이 놓여있다. 첫 번째 촛불에서 시작해 시계방향으로 $k$번째 촛불을 끄는 작업을 계속 반복한다. (이미 꺼진 촛불은 건너뛰고 숫자를 센다.) 마지막으로 남는 촛불은 몇 번인가?

### 6-1. 배열을 이용한 시뮬레이션

- 크기 $n$의 배열에 양초 번호를 저장하고, 촛불이 꺼지면 값을 `0`으로 바꾼다.
- 계속 원형으로 순회하며 $k$개를 세어야 하는데, 배열 요소가 `0`(꺼진 촛불)인지 일일이 검사하며 건너뛰어야 한다. (빈자리를 물리적으로 삭제하지 않고 놔둠)
- **결과**: 빈 공간을 계속 검사해야 하므로 비효율적이다. 한 번의 삭제에 0 체크 등 연산이 들어가며 전체 삭제까지 $O(kn)$의 시간이 걸린다.

### 6-2. 원형 연결리스트(Circular Linked List)를 이용한 시뮬레이션

- 노드들이 동그랗게 이어져 있는 원형 연결리스트를 구성한다.
- 촛불이 꺼지면 리스트에서 해당 노드를 아예 삭제(`remove`)해 버린다.
- 삭제되어 사라졌으므로, 다음 촛불을 찾을 때 이미 꺼진 촛불을 검사하거나 건너뛸 필요가 전혀 없어 매우 직관적이고 깔끔하다.
- **결과**: 한 개의 양초를 지우기 위해 링크를 $k$번 이동하고, 이 과정을 $n-1$번 지울 때까지 반복하므로 총 수행 시간은 여전히 $O(kn)$이다. (배열과 시간 복잡도는 같지만, 공간 낭비가 없고 코드와 로직이 훨씬 자연스럽다.)
