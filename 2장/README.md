# 2장. 재귀 (Recursion)

<br>

## 1\. 재귀 알고리즘이란?

알고리즘 내부에서 \*\*'자기 자신을 다시 호출'\*\*하여 문제를 해결하는 방식을 재귀적(Recursive)이라고 한다. `for`나 `while`을 사용하는 반복적(Iterative) 방식과 자주 비교된다.

- **쉬운 비유 (마트료시카 인형)**: 큰 인형을 열면 안에 똑같이 생긴 작은 인형이 있고, 그 인형을 열면 더 작은 인형이 나오는 구조와 같다. 아주 작아져서 더 이상 열 수 없는 마지막 인형이 나올 때까지 반복된다.

### 1-1. 재귀의 2가지 필수 요소

재귀 알고리즘을 짤 때는 반드시 다음 두 가지 케이스를 명확하게 분리해서 작성해야 무한 루프에 빠지지 않는다.

1.  **베이스 케이스 (Base Case / 탈출 조건)**
    - 부문제(Subproblem)가 충분히 작아져서, 더 이상 **자기 자신을 호출하지 않고 직접 정답을 반환**하며 끝내는 조건이다. (마트료시카의 가장 작은 인형)
2.  **재귀 케이스 (Recursive Case)**
    - 자기 자신을 다시 호출하는 부분이다. 이때 넘겨주는 매개변수는 반드시 원래 문제보다 \*\*'작아진(범위가 줄어든) 부문제'\*\*여야 한다.

### 1-2. 코드 예시: 1부터 $n$까지의 합 구하기 (`sum(n)`)

```c
int sum(int n) {
    // 1. 베이스 케이스 (탈출 조건)
    if (n == 1) {
        return 1;
    }
    // 2. 재귀 케이스 (더 작아진 문제로 자기 자신 호출)
    else {
        return n + sum(n - 1);
    }
}
```

## 2\. 재귀는 컴퓨터 내부에서 어떻게 작동할까? (콜 스택)

우리가 코드로 자기 자신을 부르기만 하면, 복잡한 상태 저장과 복구는 **컴퓨터(운영체제)가 알아서 처리**해 준다.

- **보류된 호출 (Suspended calls)**: `sum(3)`이 `sum(2)`를 부르면, `sum(3)`은 작업이 끝난 것이 아니라 `sum(2)`가 답을 가져올 때까지 그 자리에서 **'일시 정지(보류)'** 상태가 된다.
- **스택(Stack) 메모리 사용**: 컴퓨터는 일시 정지된 함수들의 '지역 변수'와 '돌아갈 주소(Return address)'를 스택이라는 메모리 공간에 차곡차곡 쌓아둔다.
- **작동 순서**: 베이스 케이스인 `sum(1)`에 도달해서 `1`을 반환하면, 비로소 맨 위에 쌓여있던 일시 정지된 함수들이 하나씩 깨어나면서 연쇄적으로 계산(`n + 반환값`)을 마치고 최종 답을 내놓는다.

<br>

## 3\. 재귀 알고리즘 설계의 기본 규칙

재귀를 설계할 때 반드시 지켜야 하는 원칙들이다.

1.  **탈출 조건 보장**: 모든 재귀 호출은 궁극적으로 베이스 케이스를 향해 가야 한다. (예: `n`이 점점 감소하여 결국 1이 됨)
2.  **진전 (Progress)**: 재귀 호출을 할 때마다 문제의 크기가 작아지거나 목표에 한 걸음 다가가야 한다.
3.  **믿음 (Trust the recursion)**: 내가 짠 재귀 함수가 '더 작은 문제'에 대해서는 완벽하게 작동하여 올바른 답을 가져올 것이라고 굳게 믿고 수식을 짜야 한다. (내부적으로 어떻게 도는지 머릿속으로 끝까지 추적하려 하면 너무 복잡해진다.)

## 4\. 응용 문제: 하노이의 탑 (Towers of Hanoi)

재귀 알고리즘의 꽃이라 불리는 유명한 문제다. 세 개의 말뚝(A, B, C)이 있고, A에 크기 순서대로 꽂혀있는 원반 $n$개를 전부 C로 옮기는 문제다.

- **절대 규칙**
  1.  한 번에 맨 위의 원반 한 개만 옮길 수 있다.
  2.  어떤 경우에도 작은 원반 위에 큰 원반을 올릴 수 없다.

### 4-1. 수학적 분석 (지수 시간 알고리즘)

- $n$개의 원반을 옮기기 위해서는 정확히 **$2^n - 1$회**의 이동이 필요하다.
- 만약 $n=64$개라면? $2^{64}-1 = 1.844 \times 10^{19}$회의 이동이 필요하다.
- 시간 복잡도가 $O(2^n)$인 알고리즘은 $n$이 조금만 커져도 현실적으로 해결이 불가능함을 보여주는 대표적 사례다.

### 4-2. 하노이의 탑 재귀적 해결책

복잡해 보이지만, '**가장 큰 맨 밑바닥 원반 1개**'와 '**그 위에 올려진 $(n-1)$개의 묶음**' 딱 두 덩어리로만 나눠서 생각하면 쉽게 풀린다.

1.  맨 밑의 원반을 목적지(C)로 옮기려면, 일단 그 위에 있는 **$(n-1)$개의 묶음을 방해되지 않게 보조 기둥(B)으로 치워야 한다.**
2.  이제 가장 큰 원반이 자유로워졌으니 **시작 기둥(A)에서 목적지 기둥(C)으로 옮긴다.**
3.  아까 보조 기둥(B)에 치워뒀던 **$(n-1)$개의 묶음을 목적지 기둥(C)으로 마저 옮긴다.**

<!-- end list -->

```c
// n: 옮길 원반의 수
// from: 출발 기둥 (A)
// to: 도착 기둥 (C)
// aux: 보조 기둥 (B)

void hanoi(int n, char from, char to, char aux) {
    // 1. 베이스 케이스: 원반이 1개면 그냥 바로 목적지로 옮기면 끝난다.
    if (n == 1) {
        printf("Move disk 1 from %c to %c\n", from, to);
        return;
    }

    // 2. 재귀 케이스
    // 맨 밑의 원반을 빼기 위해, (n-1)개 묶음을 먼저 보조 기둥(aux)으로 옮긴다.
    // 이때 도착 기둥(to)을 임시 보조 기둥으로 활용한다.
    hanoi(n - 1, from, aux, to);

    // 맨 밑에 남은 가장 큰 원반(n번)을 목적지 기둥(to)으로 옮긴다.
    printf("Move disk %d from %c to %c\n", n, from, to);

    // 보조 기둥(aux)에 치워뒀던 (n-1)개 묶음을 다시 목적지 기둥(to)으로 옮긴다.
    // 이때 출발 기둥(from)을 임시 보조 기둥으로 활용한다.
    hanoi(n - 1, aux, to, from);
}
```
