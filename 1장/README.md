# 1장. 알고리즘 분석과 자료구조

<br>

## 1\. 알고리즘 분석의 기초

- **알고리즘 (Algorithm)**: 주어진 문제를 유한한 시간 내에 해결하기 위한 단계적인 절차다.
- **자료구조 (Data Structure)**: 데이터를 조직하고 접근하는 체계적인 방식이다.
- **좋은 알고리즘의 척도**: 실행 시간(Running time)이 짧고, 기억 장소(Memory) 사용량이 적어야 한다.
- **최악 실행시간 (Worst-case running time)의 중요성**: 평균 실행시간은 수학적으로 구하기가 매우 까다롭다. 반면 최악의 경우는 분석이 비교적 용이하며, 게임, 재정, 로봇 등 치명적인 오류를 막아야 하는 응용 분야에서 가장 중요한 설계 기준이 되기 때문에 **알고리즘 분석은 주로 최악 실행시간에 집중**한다.

<br>

## 2\. 실행시간 측정 방법의 비교

알고리즘의 속도를 측정하는 방법에는 크게 두 가지가 있다.

### 2-1. 실험적 연구 (Experimental Studies)

직접 프로그램을 코딩하여 `clock()` 함수 등으로 실행 시간을 밀리초(ms) 단위로 측정하는 방식이다.

- **한계점**:
  1.  반드시 알고리즘을 코드로 직접 구현해야만 테스트가 가능하다.
  2.  컴퓨터의 하드웨어나 소프트웨어 환경(CPU 속도, 운영체제 등)에 따라 결과가 매번 달라진다.
  3.  모든 가능한 입력 데이터를 다 넣어볼 수 없으므로, 실험에 포함되지 않은 데이터에 대해 실행시간을 장담할 수 없다.

### 2-2. 이론적 분석 (Theoretical Analysis)

코드를 직접 짜거나 돌려보지 않고, **수학적인 방식**으로 실행시간을 분석한다.

- **장점**:
  1.  하드웨어나 소프트웨어 환경에 전혀 구애받지 않는다.
  2.  코드를 짤 필요 없이 **의사코드(Pseudocode)**만으로도 분석할 수 있다.
  3.  입력 크기 $n$에 대한 함수 형태로 실행시간을 표현하여, 모든 가능한 입력에 대해 대응할 수 있다.

<br>

## 3\. 원시작업 (Primitive Operations)

이론적 분석을 위해 코드를 구성하는 **가장 기본적이고 독립적인 계산 단계**를 '원시작업'이라고 부른다. 특정 프로그래밍 언어에 종속되지 않는 저수준(low-level) 연산들이다.

- **원시작업의 종류 (각 1번의 작업으로 셈)**
  - **수식 평가**: 덧셈, 뺄셈, 곱셈 등의 산술 연산 (`a + b`)
  - **값 할당(대입)**: 변수에 특정 값을 넣는 연산 (`i <- 1`)
  - **배열 인덱싱**: 배열의 특정 원소에 접근 (`A[i]`)
  - **메서드 호출 및 반환**: 함수를 부르거나(`myFunc()`), 값을 돌려주는 것(`return`)
  - **비교 연산**: 크기나 일치 여부를 비교 (`i < n`)

<br>

## 4. 원시작업 수 직접 계산하기

입력 데이터의 크기를 $n$이라고 할 때, 코드를 한 줄 한 줄 뜯어서 원시작업이 총 몇 번 일어나는지 계산하는 방법이다. 교안에 등장한 배열의 최댓값을 구하는 `arrayMax(A, n)` 알고리즘을 예시로 철저하게 해부해 본다.

```text
Algorithm arrayMax(A, n)
1. currentMax <- A[0]
2. i <- 1
3. while (i < n)
4.   if (A[i] > currentMax)
5.     currentMax <- A[i]
6.   i <- i + 1
7. return currentMax
```

### 원시작업 수 뜯어보기

- **Line 1: `currentMax <- A[0]`**
  - `A[0]` (배열 인덱싱 1번) + `<-` (대입 1번) = **총 2번**
- **Line 2: `i <- 1`**
  - `<-` (대입 1번) = **총 1번**
- **Line 3: `while (i < n)`**
  - `i < n` (비교 1번).
  - 주의: 이 루프 내부의 코드는 $(n-1)$번 돌지만, 조건식 검사는 루프를 끝내기 위한 마지막 실패(false) 검사까지 포함해야 하므로 **총 $n$번** 실행된다. = **총 $n$번**
- **Line 4: `if (A[i] > currentMax)`**
  - 이 코드는 루프 안에 있으므로 $(n-1)$번 반복 실행된다.
  - 1회 실행 시: `A[i]` (배열 인덱싱 1번) + `>` (비교 1번) = 2번.
  - 총 실행 횟수 = **$2(n-1)$번**
- **Line 5: `currentMax <- A[i]`**
  - **최선의 경우(Best-case)**: 첫 번째 원소가 가장 커서 한 번도 갱신되지 않으면 **0번** 실행된다.
  - **최악의 경우(Worst-case)**: 배열이 오름차순으로 정렬되어 있어 매번 최댓값이 갱신된다면 $(n-1)$번 모두 실행된다.
  - 1회 실행 시: `A[i]` (인덱싱 1번) + `<-` (대입 1번) = 2번.
  - 총 실행 횟수 (최악의 경우) = **$2(n-1)$번**
- **Line 6: `i <- i + 1`**
  - 루프 안에 있으므로 $(n-1)$번 반복 실행된다.
  - 1회 실행 시: `i + 1` (덧셈 1번) + `<-` (대입 1번) = 2번.
  - 총 실행 횟수 = **$2(n-1)$번**
- **Line 7: `return currentMax`**
  - `return` (반환 1번) = **총 1번**

### 최종 결과 계산 (최악의 경우 기준)

모든 라인의 작업 수를 더해 입력 크기 $n$에 대한 함수 $f(n)$을 도출한다.

- $f(n) = 2 + 1 + n + 2(n-1) + 2(n-1) + 2(n-1) + 1$
- $f(n) = 3 + n + 6(n-1) + 1$
- $f(n) = 3 + n + 6n - 6 + 1$
- **$f(n) = 7n - 2$**

**(참고)** 최선의 경우라면 Line 5가 0번 실행되므로: $2 + 1 + n + 2(n-1) + 0 + 2(n-1) + 1$ = $5n - 2$가 된다.

<br>

## 5\. 점근 표기법 (Asymptotic Notation)과 대문자 O ($O$)

방금 구한 $7n-2$ 같은 정확한 식은 컴퓨터마다 덧셈, 대입에 걸리는 미세한 시간이 다르기 때문에 큰 의미가 없다. 중요한 것은 데이터 크기 $n$이 무한히 커질 때 '**대략적인 증가 추세(증가율)**'가 어떠한가이다.

- **Big-Oh ($O$) 표기법**: 가장 빠르게 증가하는 '최고차항'만 남기고, 상수는 모두 버려서 추세를 단순하게 표현한다.
  - 상한(upper-bound)를 의미한다.
  - 예: $f(n) = 7n - 2$ $\rightarrow$ 상수 버리고, 차수가 가장 큰 $n$만 남겨서 $O(n)$으로 표기한다.
  - 예: $f(n) = 3n^2 + 5n + 10$ $\rightarrow$ 가장 최고차항인 $n^2$만 남겨서 $O(n^2)$으로 표기한다.

- **Big-Omega ($\Omega$) 표기법**
  - 하한(lower-bound)를 의미한다.
    
- **Big-Theta ($\Theta$) 표기법**
  - Big-Oh, Big-Omega를 모두 만족한다. 즉, 동일함수이다.
<br>
- 증가율(실행 시간) 크기 비교 요약시간 복잡도의 크기를 부등호로 표현하면 다음과 같다. <br>
    오른쪽으로 갈수록 프로그램이 느려진다.<br>
$c < \log n < \log^2 n < n < n \log n < n^2 < n^3 < 2^n$
<br>

## 6\. 응용 문제: 누적 평균 (Prefix Average) 알고리즘 비교

배열에서 $i$번째까지의 평균을 구하는 작업을 통해, 알고리즘의 효율성이 얼마나 중요한지 체감할 수 있다.

- **버전 1. 이차시간 알고리즘 $O(n^2)$**
  - $i$번째 평균을 구할 때마다 $0$번부터 $i$번까지 중첩 루프(`for`문 안의 `for`문)를 돌며 전부 다시 더한다.
  - 데이터가 늘어날수록 계산량이 제곱으로 폭증한다. 매우 비효율적이다.
- **버전 2. 선형시간 알고리즘 $O(n)$**
  - `for`문을 한 번만 돌면서, 방금 전까지 구해둔 '누적합(Running Sum)' 변수에 현재 값만 덧셈해서 평균을 구한다.
  - $n$이 커져도 계산량이 정비례해서만 늘어나므로 매우 효율적이다.
